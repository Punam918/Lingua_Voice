End-to-End Strategy: Multilingual STT + TTS Web Product
1. Project goal

Build a web app where a user can:

pick a language: en / de / es / fr / ne
click Record
get speech-to-text
type text

This should feel like a polished client demo, with a backend that can later support fine-tuned models and API deployment.

2. Best technical strategy
My recommendation
STT

Use Whisper as the primary STT base model, then fine-tune/adapt for your data, especially for Nepali and any domain-specific vocabulary. Whisper is multilingual and robust; Hugging Face’s official fine-tuning guide shows multilingual fine-tuning on Common Voice-style data.

TTS

Use a hybrid TTS stack:

English / German / Spanish / French → Coqui XTTS-v2
Nepali → MMS-TTS or a dedicated Nepali VITS-style model

This is the right move because XTTS-v2 officially supports English, Spanish, French, and German, but its published supported-language list does not include Nepali. MMS-TTS is designed for 1000+ languages and is a much safer path for Nepali TTS.

Data

Use Mozilla Common Voice as your main open-source dataset source for STT bootstrapping and multilingual coverage. Common Voice has public multilingual voice datasets and a Nepali locale page.

5. Model choices
5.1 STT model
Best baseline

Whisper small / medium / large-v3, depending your GPU budget.

Why:

multilingual
strong zero-shot baseline
fine-tuning path is well documented
practical for demo and production

OpenAI’s Whisper is multilingual and Hugging Face documents Whisper fine-tuning for multilingual ASR.

Practical choice
Development: whisper-small or whisper-medium
Production / demo: whisper-large-v3 if GPU allows
Nepali improvement: fine-tune a smaller checkpoint for Nepali/domain audio
Alternative for Nepali-heavy ASR

Use MMS ASR as a comparison baseline because MMS ASR covers 1000+ languages. That gives you a strong fallback for Nepali evaluation.

5.2 TTS model
For English, German, Spanish, French

Use Coqui XTTS-v2. Its docs list English, Spanish, French, and German among supported languages, and Coqui documents fine-tuning and dataset formatting.

For Nepali

Use MMS-TTS or a Nepali-specific open checkpoint. MMS-TTS explicitly provides text-to-speech models across 1000+ languages, which is the safer multilingual route for Nepali.

Important design decision

Do not force one TTS model for all 5 languages if one of them is unsupported.
For your project, the best architecture is:

one STT service
two TTS engines under one API

That is actually more professional.

6.2 TTS data sources

For TTS you need:

clean speech clips
exact transcripts
ideally 1 speaker per voice model
consistent sample rate
low noise
For XTTS-based tuning

Coqui documents dataset formatting and fine-tuning requirements, including custom formatter/text cleaning when needed.

For Nepali TTS

Prioritize:

clean, single-speaker Nepali recordings
normalized Devanagari text
sentence-level alignment
Data quantity targets

Practical targets:

STT fine-tune: start from 10–50 hours per language
TTS per voice: 2–10 hours clean speech can be enough for a strong portfolio system with fine-tuning, though quality improves with more curated data

The exact result depends heavily on data cleanliness; Coqui explicitly notes that model performance depends on dataset quality and hyperparameters.

7. Data preprocessing pipeline
STT preprocessing

For each audio sample:

resample to target sample rate
normalize volume
trim long silence
remove broken/transcript-mismatched files
normalize punctuation
normalize numerals
normalize whitespace
preserve language script correctly
Language-specific normalization
English/French/German/Spanish: normalize apostrophes, punctuation, accented characters
Nepali: preserve Devanagari, normalize danda/punctuation, digits, spacing
Very important

Do not over-normalize away useful orthography for French, Spanish, German, or Nepali.

17. Biggest technical risks
Risk 1: Nepali TTS quality

This is the hardest part because multilingual TTS support is uneven across models. XTTS-v2’s published language list excludes Nepali, so Nepali needs a different route such as MMS-TTS or a dedicated Nepali model.

Risk 2: Data quality

Bad alignments and noisy transcripts will ruin TTS faster than STT.

Risk 3: Browser recording variability

Microphone quality and browser codec differences can hurt STT accuracy.

Risk 4: Latency

Large models can feel slow. Use smaller checkpoints in dev and cache repeated requests.

docker compose down

cd /home/punam/LinguaVoice

Then open frontend/index.html directly in your browser OR in frontend/js/app.js change line 5:


const API = 'http://localhost:8000';
Then just open frontend/index.html as a file in your browser.
